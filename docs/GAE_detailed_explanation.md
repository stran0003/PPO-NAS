## 1. 问题设定：NAS 的信度分配困境

### 场景

Controller 分 7 步生成一个架构，最后得到一个标量奖励 R = 31.88：

```
t=0: 选 L1 层配置 → [grouped, kernel=3, init=center_spike]
t=1: 选 L2 层配置 → [1x1,    kernel=1, init=xavier]
t=2: 选 L3 层配置 → [grouped, kernel=7, init=kaiming]
t=3: 选 L4 层配置 → [LUT, spline=16]
t=4: 选 L5 层配置 → [grouped, kernel=5, init=center_spike]
t=5: 选 L6 层配置 → [1x1,    kernel=1, init=xavier]
t=6: 选 L7 层配置 → [grouped, kernel=3, init=center_spike]
                            ↓
                      R = 31.88
```

**困境**：R=31.88 是整个架构的功劳，但 7 步决策各贡献了多少？

- L3 选了 kernel=7，这个选择是好的还是坏的？
- L4 选了 spline=16，贡献了多少性能？
- L2 选了 1x1 而不是 grouped，对不对？

如果没有办法分配信度，Controller 就只能瞎猜。**GAE 就是解决这个问题的。**

## 2. 核心概念词典

在给公式之前，先把每个概念说清楚。

### 2.1 Reward（奖励）— `r`

```
定义: 执行完整个动作序列后，环境给的一个标量分数
NAS中: r = compute_reward(nmse, params)
代码:  libs/reward.py:56
形状: 标量，一个架构一个值
时机: 只在最后一步 (t=6) 给，中间步骤 r=0
```

**reward 是唯一衡量"好坏"的标尺**。NMSE 越低、参数量越少 → reward 越高 → "好架构"。

### 2.2 State Value（状态价值）— `V(s)`, values

```
定义: 从当前状态 s_t 出发，按照当前策略继续走，未来能拿到的期望累积奖励
代码:  controller.py:203 — critic(h_t)
形状: (B, 7) — 每步一个标量
来源: Critic 网络预测出来的，不是真实值
```

**V(s) 是 Critic 的猜测**——"我觉得现在的状态大概值多少钱"。比如 V(s₃) = 29，意思是 Critic 估计"从第 3 步往后走，大概能拿到 29 分的奖励"。

V(s) 是**学出来的**，一开始是随机猜测（因为 Critic 随机初始化），随着训练变得越来越准。

### 2.3 TD-Error（时序差分误差）— `δ`

```
定义: 实际拿到的奖励 + 下一状态的预估价值 − 当前状态的预估价值
公式: δ_t = r_{t+1} + γ·V(s_{t+1}) − V(s_t)
代码:  ppo.py:138/141
```

**δ 衡量"出乎意料"的程度**：

| 情况 | δ 的符号 | 含义 |
|------|----------|------|
| 实际比预期好 | δ > 0 | V(s_t) 低估了这个状态 → 这一步做得好 |
| 实际比预期差 | δ < 0 | V(s_t) 高估了这个状态 → 这一步做得差 |
| 实际和预期一致 | δ ≈ 0 | 不惊不喜，这一步正常 |

**直觉**：δ 就是"surprise"——Critic 估计这一步值 X 分，结果发现值了 X+δ 分。

### 2.4 Advantage（优势函数）— `A`

```
定义: 某个动作相对于平均水平的"超额收益"
公式: A_t = Σ_{l=0}^{∞} (γλ)^l · δ_{t+l}  (GAE)
代码:  ppo.py:140-144
```

**A_t 回答的核心问题**："选了这个动作，比平均水平好多少？"

- A_t > 0：这个动作比平均水平好 → Actor 应该多做这个动作
- A_t < 0：这个动作比平均水平差 → Actor 应该少做这个动作
- A_t ≈ 0：这个动作不好不坏

δ 是"单步 surprise"，A 是"累积 surprise"——把未来的 δ 也考虑进来。

### 2.5 Return（回报）— `R_target`, returns

```
定义: 这个状态的实际价值 = 优势 + 估值
公式: returns = A + V(s)
代码:  ppo.py:144
```

**这是 Critic 的训练标签**——Critic 输出的是 V(s)，但 GAE 告诉你"其实应该是 A+V(s)"。两者之差就是 critic_loss。

### 2.6 用日常类比理解五者关系

```
你去打游戏，打到第3关 (状态 s₃)：

V(s₃) = 30     ← Critic的猜测："我估计能得30分"
r_实际 = 35     ← 打完第3关实际得分
δ₃ = 35-30 = 5 ← "嘿，比我预期好！"（惊喜）
A₃ = Σ 未来的惊喜 ≈ 8  ← "其实我第3关的操作整体比平均好8分"
returns₃ = 8+30 = 38    ← "第3关实际值38分，不是30分"
→ Critic 学到：下次遇到类似状态，估高一点到 38
→ Actor 学到：第3关做的动作都不错，以后多做
```

### 2.7 Gamma (γ) 和 Lambda (λ)

| 参数 | 值 | 作用 | 极端值含义 |
|------|-----|------|-----------|
| γ (gamma) | 0.99 | 未来奖励的折扣率 | γ=0：只看眼前；γ=1：未来和现在一样重要 |
| λ (lambda) | 0.95 | 偏差-方差权衡 | λ=0：只用单步 TD(低方差高偏差)；λ=1：等价 Monte Carlo(无偏高方差) |

## 3. GAE 计算过程：逐步走查

### 3.1 NAS 特化：中间步骤奖励为 0

在标准 RL 中，每步都有环境给的奖励 r。但在 NAS 中，**只有最后一步有奖励**：

```
t=0   t=1   t=2   t=3   t=4   t=5   t=6
r=0   r=0   r=0   r=0   r=0   r=0   r=31.88
```

所以 GAE 公式被简化为：

```
t=6 (最后一步):  δ₆ = r_final − V(s₆)               (直接奖励 vs 估值)
                A₆ = δ₆                              (最后一步，无后续)

t<6 (中间步骤):  δ_t = γ·V(s_{t+1}) − V(s_t)          (纯价值差，没有 r)
                A_t = δ_t + γ·λ·A_{t+1}              (累积未来的优势)
```

### 3.2 一个数值实例

假设 Controller 生成了一个架构，Critic 对各步的估值为 V = [28, 29, 30, 29, 30, 29, 30]，最终奖励 R = 31.88：

```
从后往前计算：

┌─────────────────────────────────────────────────────────────────────┐
│ t=6 (L7-最后一层):                                                  │
│   δ₆ = 31.88 − 30.0 = +1.88                                       │
│   A₆ = δ₆ = +1.88                                                  │
│   解读: 最后一层的操作比 Critic 预期好 +1.88                          │
├─────────────────────────────────────────────────────────────────────┤
│ t=5 (L6):                                                           │
│   δ₅ = 0.99 × 30.0 − 29.0 = 29.7 − 29.0 = +0.7                    │
│   A₅ = 0.7 + 0.99×0.95 × 1.88 = 0.7 + 1.77 = +2.47                │
│   解读: 这一步状态变好了(δ>0)，而且未来还有正优势传过来               │
├─────────────────────────────────────────────────────────────────────┤
│ t=4 (L5):                                                           │
│   δ₄ = 0.99 × 29.0 − 30.0 = −0.29                                  │
│   A₄ = −0.29 + 0.94 × 2.47 = +2.03                                 │
│   解读: 一步来看状态稍微变差了(δ<0)，但继承了前面的正优势              │
├─────────────────────────────────────────────────────────────────────┤
│ t=3 (L4-LUT):                                                       │
│   δ₃ = 0.99 × 30.0 − 29.0 = +0.7                                   │
│   A₃ = 0.7 + 0.94 × 2.03 = +2.61                                   │
├─────────────────────────────────────────────────────────────────────┤
│ t=2 (L3):                                                           │
│   δ₂ = 0.99 × 29.0 − 30.0 = −0.29                                  │
│   A₂ = −0.29 + 0.94 × 2.61 = +2.16                                 │
├─────────────────────────────────────────────────────────────────────┤
│ t=1 (L2):                                                           │
│   δ₁ = 0.99 × 30.0 − 29.0 = +0.7                                   │
│   A₁ = 0.7 + 0.94 × 2.16 = +2.73                                   │
├─────────────────────────────────────────────────────────────────────┤
│ t=0 (L1):                                                           │
│   δ₀ = 0.99 × 29.0 − 28.0 = +0.71                                  │
│   A₀ = 0.71 + 0.94 × 2.73 = +3.28                                  │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 观察

**所有 A_t 都 > 0**：因为这个架构最终拿到了好奖励（R=31.88 > 所有 V(s)），**所有步骤都"沾了光"**。

**越前面的步优势越大**（A₀=3.28 > A₆=1.88）：因为 γλ=0.94 的指数衰减，优势从前向后指数递减。但这个衰减很慢（0.94⁶=0.69），所以早期步骤也能分到相当的优势。

**这是合理的**：找到了一个好架构 → 所有参与决策的步骤都应该得到正反馈。但如果 Critic 在某个状态估得很高而下一步实际没涨那么多，δ 就为负，那一步的贡献就会被削弱。

## 4. 为什么这样计算：从 TD 到 GAE

### 4.1 三种可能的信度分配方式

**方式一：给每步相同的奖励（均分）**

```
A₀ = A₁ = ... = A₆ = 31.88 / 7 = 4.55
```

问题：第 2 步选了 1x1（合路层），和第 3 步选了 grouped+kernel=7（高质量分组卷积），得到的反馈一模一样。Controller 无法区分哪个决策真正有价值。

**方式二：纯 TD（只看当前步的 δ）**

```
A_t = δ_t = γ·V(s_{t+1}) − V(s_t)
```

问题：只看一步的"惊喜"，看不到"第 6 步的操作让奖励暴涨 +1.88"这个全局事实。短视。

**方式三：Monte Carlo（看最终奖励）**

```
A_t = r_final − V(s_t)    （无折扣时）
```

问题：真正 unbiased（因为用了真实奖励），但方差极大——如果这个架构恰好评估时噪声大（nmse 测量有波动），所有步骤的优势都跟着剧烈波动。

### 4.2 GAE 是折中

```
A_t^GAE = δ_t + γλ·δ_{t+1} + (γλ)²·δ_{t+2} + ... + (γλ)^{T-t-1}·δ_{T-1}
```

| λ 值 | 等价于 | 特点 |
|------|--------|------|
| λ=0 | A_t = δ_t（纯 TD） | 低方差，高偏差（只看眼前） |
| λ=1 | A_t ≈ r_final − V(s_t)（MC） | 无偏，高方差（受噪声影响大） |
| λ=0.95 | 接近 MC，但用指数衰减平滑远期噪声 | 平衡 |

λ=0.95 意味着：当前步的 δ 权重=1，下一步权重=0.94，再下一步=0.88... 越远的 δ 影响越小。这保证了"最终奖励 31.88"的信息能传回第一步，同时不会被评估噪声过度影响。

### 4.3 为什么用 V(s) 做 baseline

直接用 reward 做优势有一个问题：

```
假设所有架构的平均 reward = 25
架构 A 拿到 31.88 → "太好了，应该多做"
架构 B 拿到 26.00 → "也不错，也要多做"

但实际上 26.00 只比平均好一点点，不值得大举跟进
```

V(s) 提供了**依赖上下文的 baseline**：

```
架构 B 在第 3 步时 V(s₃)=30 → 结果最终 reward=26 → δ₆=-4
→ "别看前面觉得不错，最后搞砸了 → 前面的决策要反思"
```

这就是为什么 GAE 比直接用 reward 更精细——它把"这个状态本来能拿多少"作为参照系，只关心**超过预期的部分**。

### 4.4 标准化

[ppo.py:147](libs/ppo.py#L147)：

```python
advantages = (A - mean(A)) / (std(A) + 1e-8)
```

标准化后，整个批次中：一半步骤的 A 为正（"你做对了"），一半为负（"你做错了"）——即使用的是上述全正 A 的例子，标准化后也会变成有正有负。

这不是改变信息，而是让梯度更稳定——正负优势的数量平衡，避免所有样本都在"鼓励"方向导致策略跑偏。

## 5. 奖励函数如何决定"对错"

### 5.1 奖励函数 = 判断标准

```python
def compute_reward(nmse, num_params, config):
    perf_score = -nmse * 1.0             # NMSE越小 → 得分越高
    param_penalty = 0.1 * (num_params / 100000)
    return perf_score - param_penalty
```

**这个函数定义了"好架构"的标准**：

| R 高（>30） | nmse 好（<−30）+ params 合理（<100K） | "好架构" → A_t > 0 → 多做一些 |
| R 低（<25） | nmse 差（>−25）或 params 太大（>200K） | "差架构" → A_t < 0 → 少做一些 |

### 5.2 奖励如何传入 GAE

```
evaluate_architecture(arch) → nmse, params
    ↓
compute_reward(nmse, params) → r (标量)
    ↓
16个r组成 rewards (16,)
    ↓
_compute_gae(rewards, values)
    ↓
A_t 和 returns
```

**所有关于"好坏"的信息都压缩进了 16 个标量 r**，然后 GAE 把它们展开成 16×7=112 个优势值（对每一步的指导信号）。

### 5.3 奖励函数设计的实际影响

假设有两个架构：

```
架构 X: nmse=-35, params=150000  → R = 35 - 0.15 = 34.85  (极好)
架构 Y: nmse=-28, params=80000   → R = 28 - 0.08 = 27.92  (一般)
```

如果 param_weight 从 0.1 改成 0.5：

```
架构 X: R = 35 - 0.75 = 34.25  (仍然好)
架构 Y: R = 28 - 0.40 = 27.60  (变化不大)
```

但如果架构 Z 是 nmse=-32, params=300000：

```
param_weight=0.1: R = 32 - 0.3 = 31.70   (相当好)
param_weight=0.5: R = 32 - 1.5 = 30.50   (没那么好了)
```

**参数权重的选择直接影响了"参数量大的好架构"是否值得追**。如果设太高，Controller 会倾向于生成小模型（参数少），即使性能略差；如果设太低，Controller 会无脑堆参数。

## 6. 从 GAE 到梯度：模型如何调整

### 6.1 完整的数据流

```
架构评估 → nmse, params
    ↓                        ↓
compute_reward()       用于 Pareto 前沿记录
    ↓
rewards (16,)
    ↓
_compute_gae(rewards, values)   ← Critic 的 V(s) 作为 baseline
    ↓
advantages (16,7)  ← "每一步有多大功劳"
returns (16,7)     ← "每一步实际值多少"
    ↓
_ppo_update():
  for epoch in 1..4:
    for minibatch:
      evaluate(batch) → new_log_probs, new_values, H
      ratio = exp(new_log_probs − old_log_probs)
      actor_loss = −mean(min(ratio×A, clip(ratio)×A))
      critic_loss = MSE(new_values, returns)
      loss = actor_loss + 0.5×critic_loss − 0.01×H
      loss.backward()    ← 梯度流遍整个 Controller
      optimizer.step()
```

### 6.2 Actor 如何"知道往哪个方向调"

以第 3 步（L3 层）为例，假设两个样本在这一步都选了 grouped+kernel=3，但最终一个拿到了 A=+2.16（好）另一个拿到了 A=−1.5（差）：

```
样本 1 (好): ratio_old=1.0, A=+2.16
  → actor_loss 对 log_prob 的梯度方向: "多做这个动作"
  → gradient 负的（因为 loss = -mean(...)）

样本 2 (差): ratio_old=1.0, A=-1.5
  → actor_loss 对 log_prob 的梯度方向: "少做这个动作"
  → gradient 正的

两个样本的梯度在 minibatch 里加起来 → Actor 头的参数被往"增加 P(grouped,k=3)"的方向推
→ 下一轮采样时，Controller 更倾向于选 grouped+kernel=3 这个组合
```

### 6.3 Critic 如何变得准确

```
returns₃ = A₃ + V₃ = 2.16 + 30 = 32.16   ← "L3 这个状态实际值 32.16"

但是 V_new(L3) 预测了 29.5 → critic_loss = MSE(29.5, 32.16)
→ backward → Critic 头参数调整 → 下次类似状态 V 会估高一点

随着训练进行，V(s) 越来越准 → GAE 的优势估计越来越可靠 → Actor 学得越来越好
```

### 6.4 三者的互促循环

```
        ┌──────────────────────────────────┐
        │                                  │
        ▼                                  │
  Actor 采样更好的架构                       │
        │                                  │
        ▼                                  │
  评估得到更高的奖励 → GAE 优势更准            │
        │                                  │
        ▼                                  │
  Critic 学到更好的 V(s) ────────────────────┘
        │
        ▼
  GAE 优势更可靠
        │
        ▼
  Actor 继续改进
```

## 7. 完整数值走查

以一个具体架构从采样到梯度更新的全过程：

### 7.1 架构和评估

```
架构: L1(grouped,k=3,cs) → L2(1x1,k=1,x) → L3(grouped,k=7,kai) → L4(LUT,sp=16)
     → L5(grouped,k=5,cs) → L6(1x1,k=1,x) → L7(grouped,k=3,cs)

评估: nmse = -32.5, params = 110000
奖励: R = 32.5 - 0.11 = 32.39
```

### 7.2 GAE 计算 (V 为假设值)

```
步    V(s)       r       δ                A (累积)          returns=A+V
─────────────────────────────────────────────────────────────────────────
t=6   29.5    32.39    +2.89             +2.89             32.39  ← 对上了!
t=5   29.0      0     0.99×29.5-29.0    +0.205+0.94×2.89   32.92
                               =+0.205    =+2.92             
t=4   29.5      0     0.99×29.0-29.5    -0.29+0.94×2.92   32.74
                               =-0.29     =+2.46            
t=3   28.5      0     0.99×29.5-28.5    +0.705+0.94×2.46  31.52
                               =+0.705    =+3.02            
t=2   29.0      0     0.99×28.5-29.0    -0.285+0.94×3.02  31.48
                               =-0.285    =+2.55            
t=1   28.0      0     0.99×29.0-28.0    +0.71+0.94×2.55   31.04
                               =+0.71     =+3.11            
t=0   27.5      0     0.99×28.0-27.5    +0.22+0.94×3.11   30.95
                               =+0.22     =+3.14            
```

**关键验证**：t=6 的 returns = 32.39 = reward，这是 GAE 正确性的一个标志（最后一步的 return 应该等于实际奖励）。

### 7.3 Actor 如何被训练

标准化后，每个优势值变成相对于批次均值的 z-score。然后：

```
第 2 步 (L3层): 选了 grouped+kernel=7, 这一步是"好决策"
  → A₂ > 0 (标准化后)
  → actor_loss 的梯度方向: 增加 P(grouped) 和 P(kernel=7)

第 4 步 (L5层): 虽然也选了 grouped+kernel=5, 但这一步的 δ₄=−0.29
  → 说明前面 V(s₄)=29.5 给高了, 这一步的状态不如预期
  → A₄ 虽然 >0 (因为继承了后面的正优势), 但比 A₃ 小
  → 梯度也会增加 P(grouped,k=5), 但"推力"比 t=2 弱

第 1 步 (L2层): 选了 1x1 → log_prob 里没有 kernel 项
  → A₁=+3.11 → 对"选 1x1"这个决定本身是正反馈
  → 但不会影响到 kernel 头的参数（因为 1x1 不采样 kernel）
```

### 7.4 Critic 如何被训练

```
t=2:  V_new(s₂)=28.3 → returns=31.48 → MSE → 更新 Critic, 下次估高一点
t=3:  V_new(s₃)=28.2 → returns=31.52 → MSE → 更新 Critic, 下次估高一点
t=4:  V_new(s₄)=29.8 → returns=32.74 → MSE → 更新 Critic, 这次估得不够高
```

**Critic 逐渐学会**：L4 (LUT层) 之后的状态通常比 L4 之前的状态更有价值（因为越往后越接近最终奖励）。

---

## 8. 总结：全链路追踪

```
┌─────────────────────────────────────────────────────────────────┐
│               从"好架构"到"参数更新"的完整链路                    │
│                                                                 │
│  评估得到 nmse=−32.5, params=110000                              │
│       │                                                         │
│       ▼                                                         │
│  compute_reward: R = 32.5 − 0.11 = 32.39                        │
│  → "32.39 分，是个好架构"                                       │
│       │                                                         │
│       ▼                                                         │
│  GAE: 把 32.39 这个总分 沿 7 步时间轴反向分配                    │
│  A₀=3.14, A₁=3.11, ..., A₆=2.89                                 │
│  → "第0步贡献了3.14，第3步贡献了3.02..."                         │
│       │                                                         │
│       ▼                                                         │
│  标准化: (A − mean_batch) / std_batch                            │
│  → 批次内比较，A>0的步→应多做，A<0的步→应少做                    │
│       │                                                         │
│       ▼                                                         │
│  actor_loss = −mean(min(ratio×A, clip(ratio)×A))                │
│  → 对 A>0 的步，梯度推动 P(该动作)↑                              │
│  → 对 A<0 的步，梯度推动 P(该动作)↓                              │
│       │                                                         │
│       ▼                                                         │
│  loss.backward() → 梯度流过 Actor头、Critic头、共享LSTM          │
│  → Controller 学会了"怎样的架构设计能拿到高分"                   │
│       │                                                         │
│       ▼                                                         │
│  下一轮采样：Controller 更倾向于生成高奖励架构的特征              │
└─────────────────────────────────────────────────────────────────┘
```

**最终答案**：模型不是直接被告知"第 3 步选 grouped 是对的"，而是通过 **GAE 将终端奖励展开成每个时间步的优势信号**，再通过 **PPO 的 actor_loss** 将优势信号转化为梯度，推动 Actor 头的参数往"多做优势大的动作、少做优势小的动作"的方向调整。整个过程是全自动的——你只需要定义好 `compute_reward()` 这个"什么算好"的标准，GAE+PPO 会自动解决"怎么变好"的问题。
