1. 问题设定：NAS 的信度分配困境

场景

Controller 分 7 步生成一个架构，最后得到一个标量奖励 R = 31.88：

t=0: 选 L1 → [grouped, kernel=3, init=center_spike]
t=1: 选 L2 → [1x1,    kernel=1, init=xavier]
t=2: 选 L3 → [grouped, kernel=7, init=kaiming]
t=3: 选 L4 → [LUT, spline=16]
t=4: 选 L5 → [grouped, kernel=5, init=center_spike]
t=5: 选 L6 → [1x1,    kernel=1, init=xavier]
t=6: 选 L7 → [grouped, kernel=3, init=center_spike]

                      ↓
                R = 31.88  ← 整个架构的功劳，但 7 步各贡献了多少？
困境：L3 选了 kernel=7，这是好是坏？L4 选了 spline=16，贡献了多少？L2 选了 1x1 是否正确？
如果没有办法分配信度，Controller 只能瞎猜。GAE 就是解决这个问题的。

2. 核心概念词典

2.1 五个核心概念的日常类比

你去打游戏，打到第 3 关（状态 s₃）：

概念类比符号值
Critic 估值"我猜第3关大概能拿30分"V(s₃)30
实际奖励"实际拿到了35分"r35
TD-Error"嘿，比我预期多了5分！"（惊喜）δ₃+5
Advantage"仔细一想，我第3关的操作整体比平均好8分"A₃+8
Return"所以第3关实际值38分，不是30分"returns₃38

→ Critic 学到：下次估高一点到 38
→ Actor 学到：第 3 关做的动作不错，以后多做

2.2 精确定义

Reward (r) — 奖励

定义：执行完整个动作序列后，环境给的标量分数
NAS 中：r = compute_reward(nmse, params)
时机：只在最后一步给，中间步骤 r=0
地位：唯一衡量"好坏"的标尺

State Value V(s) — 状态价值

定义：从当前状态 s_t 出发，按当前策略走，未来能拿到的期望累积奖励
NAS 中：V(s_t) = critic(h_t) — Critic 网络预测的标量
本质：Critic 的猜测——"我觉得现在大概值多少钱"
关键：一开始是随机猜测，随着训练越来越准

TD-Error (δ) — 时序差分误差

公式：δ_t = r_{t+1} + γ·V(s_{t+1}) − V(s_t)
含义："出乎意料"的程度
δ > 0 → V(s_t) 低估了 → 这一步比预期好
δ < 0 → V(s_t) 高估了 → 这一步比预期差
代码：ppo.py:138/141

Advantage (A) — 优势函数

公式：A_t = Σ(γλ)^l · δ_{t+l} (GAE 累积)
含义：某个动作相对于平均水平的 "超额收益"
A > 0 → 这个动作比平均水平好 → Actor 应该多做
A < 0 → 这个动作比平均水平差 → Actor 应该少做
代码：ppo.py:143-144

Return / R_target — 回报

公式：returns = A + V(s)
含义：这个状态的 "实际价值" = 估值 + 超额
用途：Critic 的训练标签——"你的估值应该是这个，不是原始 V(s)"
代码：ppo.py:147

γ 和 λ

参数值作用极端值
γ (gamma)0.99未来奖励的折扣率0=短视, 1=远视
λ (lambda)0.95偏差-方差权衡0=纯TD, 1=纯MC

3. GAE 计算过程：逐步走查

3.1 NAS 特化：中间步骤奖励为 0

标准RL: r_0, r_1, r_2, r_3, r_4, r_5, r_6  (每步都有)
NAS:    0,   0,   0,   0,   0,   0, 31.88  (只有最后一步有)

所以 GAE 公式被简化为：

t=6:  δ₆ = r_final − V(s₆)           直接奖励 vs 估值
     A₆ = δ₆

t<6:  δ_t = γ·V(s_{t+1}) − V(s_t)      纯价值差
     A_t = δ_t + γ·λ·A_{t+1}          累积未来的优势

3.2 数值走查

假设 V = [28, 29, 30, 29, 30, 29, 30]，R = 31.88：

t=6 (L7): δ₆ = 31.88 − 30.0 = +1.88 | A₆ = +1.88
最后一层的操作比 Critic 预期好
t=5 (L6): δ₅ = 0.99×30.0 − 29.0 = +0.70 | A₅ = 0.70 + 0.94×1.88 = +2.47
状态变好了，而且未来有正优势传过来
t=4 (L5): δ₄ = 0.99×29.0 − 30.0 = −0.29 | A₄ = −0.29 + 0.94×2.47 = +2.03
一步来看状态变差了，但继承了前面的正优势
t=3 (L4): δ₃ = +0.70 | A₃ = +2.61
t=2 (L3): δ₂ = −0.29 | A₂ = +2.16
t=1 (L2): δ₁ = +0.70 | A₁ = +2.73
t=0 (L1): δ₀ = +0.71 | A₀ = +3.28
所有步优势都>0 → 每个决策步都沾了"好架构"的光

关键观察

所有 A_t 都 > 0：因为最终奖励 31.88 高于所有 V(s)，好消息被整条链共享。

A₀ > A₆：早期步骤也分到了充分的优势（γλ=0.94 衰减很慢，0.94⁶=0.69）。

δ₄=−0.29 但 A₄=+2.03：一步不好没关系，后面的好结果救了它。

3.3 标准化

A = (A − mean_batch(A)) / (std_batch(A) + 1e-8)   ppo.py:147

标准化后，批次内一半 A 为正（"做对了"），一半为负（"做错了"）——让正负梯度信号平衡，训练更稳定。

4. 为什么这样计算：从 TD 到 GAE

4.1 三种信度分配方式的对比

方式做法优点致命问题
均分 Aₜ = R/7 简单 选 1x1 和选 grouped 拿到同样的反馈 → Controller 无法区分哪些决策真正有价值
纯 TD (λ=0) Aₜ = δₜ 低方差 只看眼前 surprise，看不到"第6步让奖励暴涨"这个全局事实 → 短视
Monte Carlo (λ=1) Aₜ = R − V(sₜ) 无偏 方差极大——nmse测量噪声全传导到每一步 → 训练振荡
GAE (λ=0.95) Aₜ = Σ(γλ)ˡ·δₜ₊ₗ 平衡 信息传到底 + 噪声被指数衰减平滑

4.2 λ 如何控制权衡

A₀ = δ₀ + (γλ)·δ₁ + (γλ)²·δ₂ + (γλ)³·δ₃ + (γλ)⁴·δ₄ + (γλ)⁵·δ₅ + (γλ)⁶·δ₆
      权重=1   权重=.94  权重=.88  权重=.83  权重=.78  权重=.73  权重=.69

λ=0.95 的精妙之处：越远的 δ 权重越小，但衰减很慢。第六步的 δ₆ 仍然有 69% 的权重传到第一步 —— 既保证了"最终奖励是好是坏"这个信息能被所有步骤接收到，又不会让评估噪声完全主导。

4.3 为什么用 V(s) 做 baseline 而不是直接用 reward

架构 A: R=31.88, 架构 B: R=26.00
直接用 R: A 每步都正反馈, B 每步都正反馈 → 都"鼓励"
用 GAE:  A 的 V(s)≈28~30, R=31.88 → A>0 ✅
         B 的 V(s)≈28~30, R=26.00 → A<0 ❌ (估值给高了, 最终没达到)

V(s) 提供了依赖上下文的 baseline——"这个状态本来能拿多少"是参照系，只有超过预期的部分才算优势。

5. 奖励函数如何决定"对错"

5.1 奖励函数 = "好坏"的唯一定义

def compute_reward(nmse, num_params, config):
    perf_score = -nmse * 1.0              # NMSE越小 → 得分越高
    param_penalty = 0.1 * (num_params / 100000)
    return perf_score - param_penalty
R 值对应架构判定→ GAE 效应
>30nmse好 + params合理好架构A>0 → Actor多做这类的决策
<25nmse差 或 params太大差架构A<0 → Actor少做这类的决策

5.2 param_weight 如何影响搜索方向

架构 Z: nmse=−32, params=300000
param_weight=0.1 → R = 32 − 0.3 = 31.70 (相当好，会追)
param_weight=0.5 → R = 32 − 1.5 = 30.50 (没那么好，不追了)

参数权重的选择直接影响了"大模型是否值得追"
太高 → Controller 倾向小模型（性能略差）；太低 → 无脑堆参数

5.3 奖励如何转化为对每一步的"对错判定"

评估
nmse, params
→
compute_reward

→
GAE

(16×7)

所有关于"好坏"的信息都压缩进了 16 个标量 r，然后 GAE 把它们展开成 16×7=112 个对每一步的指导信号。

6. 从 GAE 到梯度：模型如何调整

6.1 Actor 怎么"知道往哪个方向调"

好样本 (A > 0)

第3步选了 grouped+kernel=3
A₃ = +2.16

actor_loss 对 log_prob
的梯度方向: "多做这个动作"

→ P(grouped,k=3) ↑

差样本 (A < 0)

第3步选了 grouped+kernel=9
A₃ = -1.50

actor_loss 对 log_prob
的梯度方向: "少做这个动作"

→ P(grouped,k=9) ↓

一个 batch 里好样本和差样本的梯度加在一起 → Actor 头参数往"增加好动作概率、减少差动作概率"的方向移动。

6.2 Critic 怎么变准

returns₃ = A₃ + V₃ = 2.16 + 30 = 32.16   ← "L3 实际值 32.16"
V_new(L3) = 29.5                         ← "Critic 猜了 29.5"

critic_loss = MSE(29.5, 32.16) → backward → Critic头参数调整
→ 下次遇到类似状态 V 会估高一点

核心答案：模型不是直接被告知"第 3 步选 grouped 是对的"，而是通过
奖励函数（定义"什么算好"）→ GAE（将总分沿时间轴分解为每步的优势）→ PPO（将优势转化为梯度）
三层机制接力，最终实现"Controller 自动学会生成高奖励架构"。
